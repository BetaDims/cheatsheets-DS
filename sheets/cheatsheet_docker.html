antes de los comandos

¿Qué es Docker?

B

Docker empaqueta una aplicación con todas sus dependencias en una imagen portable que corre igual en cualquier máquina. Un contenedor es esa imagen en ejecución: un proceso aislado que comparte el kernel del host, mucho más ligero que una máquina virtual.

Contenedorproceso aislado (CPU, red, FS propios) que comparte el kernel del host.
VMSO invitado completo sobre un hipervisor: pesada, arranque en minutos.
Contenedor ≈arranca en milisegundos, ocupa MB, no GB.
La idea clave"Funciona en mi máquina" deja de ser excusa: la imagen lleva dentro runtime, librerías y código.

Arquitectura

I

La CLI (docker) habla con el daemon (dockerd), que construye y corre contenedores y tira imágenes de un registry (Docker Hub por defecto).

clienteel comando docker que escribes.
daemonservicio que hace el trabajo real.
registryalmacén de imágenes (Docker Hub, GHCR…).
objetosimágenes, contenedores, volúmenes, redes.

Imagen vs contenedor — capas de solo lectura

B

Una imagen son capas de solo lectura apiladas (cada instrucción del Dockerfile es una capa cacheable). Al ejecutar docker run se añade encima una capa de escritura efímera: eso es el contenedor. Borra el contenedor y esa capa se va; la imagen no cambia.

docker image · capas

Anatomía: del Dockerfile al contenedor en marcha

I

El recorrido completo: el Dockerfile es la receta; docker build la cuece en una imagen (plantilla inmutable); docker run la enciende como contenedor (proceso efímero). Para que los datos sobrevivan usas un volumen; para que los contenedores se hablen, una red.

Verificar la instalación

B

Comprueba que la CLI ve al daemon y lanza el contenedor de prueba. En Windows/Mac, Docker Desktop arranca el daemon; en Linux, el servicio docker.

Permisos en LinuxAñade tu usuario al grupo docker (sudo usermod -aG docker $USER) para no escribir sudo en cada comando. Cierra y reabre sesión.

Obtener & listar imágenes

B

Las imágenes se nombran repo:tag. Sin tag, Docker asume :latest (no fijes producción a latest).

tag = versiónUsa tags concretos (node:20-alpine) para builds reproducibles.

Construir desde un Dockerfile

I

build lee el Dockerfile del contexto (el . final) y produce una imagen etiquetada con -t.

El contexto importaEl . es el contexto de build: todo se envía al daemon. Usa .dockerignore para no enviar node_modules ni .git.

Etiquetar & publicar en un registry

I

Para subir a un registry, la imagen debe llevar el prefijo del registry/usuario. tag crea un alias; push la sube tras login.

docker run — el comando estrella

B

run = crear + arrancar un contenedor desde una imagen. Sus flags resuelven el 90% de los casos:

--namenombre legible al contenedor

-vmonta volumen / carpeta
--rmauto-borrar al salir
--restartpolítica de reinicio

Ver, entrar y depurar

B

logs y exec son tus ojos dentro del contenedor en marcha.

Ciclo de vida: parar & borrar

B

Parar no borra: el contenedor sigue existiendo (y ocupa espacio) hasta que lo eliminas con rm.

Estado = efímeroTrata los contenedores como desechables: todo dato que deba sobrevivir va en un volumen (sección 04), nunca en la capa de escritura.

Dockerfile real (Node)

I

Copia primero el manifiesto de dependencias y luego el código: así Docker cachea el npm install mientras solo cambie tu código.

CMD vs ENTRYPOINT

A

El más confuso. Regla práctica:

CMDargumentos por defecto, fáciles de sobrescribir en docker run.
ENTRYPOINTel ejecutable fijo; lo que escribas en run se le pasa como argumentos.
JuntosENTRYPOINT = binario, CMD = sus flags por defecto.

Multi-stage build — imágenes diminutas

A

Compila en una etapa pesada (con compiladores) y copia solo el resultado a una imagen final mínima. La imagen de producción no carga las herramientas de build.

Orden = cachéCada instrucción es una capa cacheada. Pon lo que menos cambia arriba (deps) y tu código abajo: así un cambio de código no invalida la instalación de dependencias.

Datos & redes

Volúmenes (datos persistentes)

I

Un volumen lo gestiona Docker y sobrevive a la muerte del contenedor. Es la forma recomendada de guardar bases de datos.

Bind mounts (carpeta del host)

I

Un bind mount mapea una carpeta de tu máquina dentro del contenedor: ideal en desarrollo para ver cambios al instante.

Volumen vs bindVolumen = datos de producción gestionados por Docker. Bind = tu código en desarrollo.

Redes — contenedores que se hablan

I

Crea una red bridge propia y conecta varios contenedores: dentro de ella se resuelven por nombre (DNS interno), sin exponer puertos al host.

DriverCuándo
bridgepor defecto; red privada entre contenedores
hostcomparte la red del host (sin aislar)
nonesin red

I

Limpieza & buenas prácticas

recuperar disco
y producción sólida

prune — recuperar espacio

B

Docker acumula imágenes, contenedores parados, volúmenes y caché de build. prune los limpia.

Cuidadovolume prune y system prune -a --volumes borran datos. Revisa antes de ejecutar en un servidor con bases de datos.

inspect · stats · cp

I

inspect vuelca toda la config en JSON; con --format extraes un campo.

Checklist de buenas prácticas

A
Imágenes finasparte de -slim / alpine y usa multi-stage. Menos peso = menos superficie de ataque.
.dockerignoreexcluye .git, node_modules, secretos y artefactos del contexto.
No-rootañade USER no privilegiado; nunca corras como root sin motivo.
Tags fijosfija versiones (:1.27) en producción; :latest rompe builds reproducibles.
Sin secretosnada de claves en el Dockerfile (quedan en las capas); usa variables/secrets en runtime.
Un procesoun contenedor = una responsabilidad; orquesta con Compose, no metas todo dentro.
Caché de capascopia dependencias antes que el código para reusar la caché del build.
Healthcheckdefine HEALTHCHECK para que el orquestador sepa si reiniciar.
