¿Qué es un agente?

B

Un agente es un LLM que, en un bucle, decide qué herramientas usar para avanzar hacia un objetivo, observa el resultado y repite hasta terminar. La diferencia con un simple chatbot es la acción: el agente hace, no solo responde.

Autonomíaelige sus pasos; no sigue un guion fijo.
Herramientassu forma de actuar en el mundo (APIs, código, archivos).
Buclepensar → actuar → observar, hasta el objetivo.

Flujo de trabajo vs agente

I

No todo necesita un agente. Si los pasos son fijos, un flujo de trabajo (workflow) es más simple, barato y predecible.

Quién decide los pasos
Workflowtú, en código (rutas fijas)
Agenteel modelo, en tiempo de ejecución
ReglaEmpieza por lo más simple. Sube a agente solo cuando la tarea es abierta y difícil de guionizar.

¿Deberías construir un agente?

I

Cuatro preguntas antes de lanzarte. Si alguna es "no", quédate en un flujo más simple:

Complejidad¿la tarea es multi-paso y difícil de especificar de antemano?
Valor¿el resultado justifica más coste y latencia?
Viabilidad¿el modelo es capaz de hacerlo bien?
Coste del error¿puedes detectar y recuperar fallos (tests, revisión)?

Las cuatro piezas

B

Un agente combina un modelo (el cerebro que razona y decide), herramientas (cómo actúa), memoria/contexto (lo que sabe) y un bucle que lo orquesta sobre un entorno.

anatomía · modelo + herramientas + memoria + bucle

El bucle — el corazón

I

El runtime (tu código, o un framework) ejecuta el bucle: pide al modelo, si pide una herramienta la ejecuta y le devuelve el resultado, y repite hasta que el modelo deja de pedir herramientas. El modelo decide; tu harness ejecuta.

Tool use (function calling)

Definir una herramienta

I

Una herramienta = nombre + descripción + esquema JSON de sus argumentos. La descripción es lo que el modelo lee para decidir cuándo usarla: sé claro y di cuándo llamarla.

El bucle manual (Anthropic)

A

Control fino: llamas a la API, si stop_reason es tool_use ejecutas la tool y devuelves un tool_result, repitiendo hasta end_turn.

Tool runner — el bucle, automático

I

El SDK puede llevar el bucle por ti: defines funciones tipadas como tools y el tool runner se encarga de llamar, ejecutar y reinyectar resultados hasta terminar.

Qué es MCP

I

El Model Context Protocol es un estándar abierto (como "USB-C para LLMs") que conecta agentes con herramientas y datos. Un servidor MCP expone capacidades; un cliente (tu agente) las consume — sin pegamento a medida por cada integración.

ExponeQué es
toolsacciones que el modelo puede ejecutar
resourcesdatos/archivos que puede leer
promptsplantillas de prompt reutilizables

Un servidor MCP mínimo

A

Con el SDK de Python, una función decorada se convierte en una tool que cualquier cliente MCP (Claude Code, Cowork…) puede usar.

Por qué importaEscribes la integración una vez y sirve para todos los clientes MCP. Es lo que hace los agentes interoperables.

Contexto: skills & memoria

lo que el agente

Skills

I

Una skill es una carpeta con un SKILL.md que aporta instrucciones especializadas bajo demanda: su descripción está siempre en contexto, el detalle se carga solo cuando la tarea encaja (revelación progresiva). Mantiene el prompt base pequeño.

Para quéflujos, convenciones y best-practices reutilizables.
vs herramientala skill enseña cómo; la tool ejecuta.
vs promptla skill se carga sola; el prompt lo pones tú cada vez.

Memoria & gestión de contexto

A

La ventana de contexto es finita. Los agentes largos la cuidan con varias técnicas:

Memoriaguarda estado en archivos para sobrevivir entre sesiones.
Compactaciónresume el historial al acercarse al límite.
Context editingpoda resultados de tools viejos e irrelevantes.
RAGrecupera solo los fragmentos relevantes en vez de meterlo todo.
Context engineeringTan importante como el prompt: decide qué entra en contexto y cuándo, no solo cómo lo pides.

PatrónIdea

Evaluatorun paso genera, otro critica y mejora

Single vs multi-agente

A

Un orquestador puede delegar subtareas a subagentes especializados, cada uno con su contexto y herramientas. Útil para paralelizar o aislar trabajo; añade complejidad y coste.

Orchestratorcoordina y reparte; reúne resultados.
Subagentescada uno experto en una parte (revisar, buscar).
FrameworksClaude Agent SDK, OpenAI Agents SDK y similares dan el andamiaje.
CuándoEmpieza con un solo agente. Pasa a multi-agente solo si hay trabajos independientes que paralelizar o aislar.

Evaluar & operar

Evals

I

Sin medición no hay mejora. Define un conjunto de casos con resultado esperado y mide cada cambio contra él, en vez de fiarte de "se ve bien".

Datasetcasos representativos con su respuesta correcta.
Métricaéxito de tarea, no solo "suena bien".
LLM-as-judgeun modelo evalúa la salida con una rúbrica.

Seguridad & coste

A

Un agente actúa: ponle límites antes de soltarlo.

Permisospide confirmación en acciones irreversibles (borrar, enviar, pagar).
Aislamientoejecuta herramientas en sandbox; nada de secretos al alcance.
Límitestope de iteraciones y de tokens para no quemar coste.
Observabilidadregistra cada paso (trazas) para depurar y auditar.
Inyección de promptsTrata todo dato externo como potencialmente hostil: puede intentar redirigir al agente. Valida en los límites del sistema.
