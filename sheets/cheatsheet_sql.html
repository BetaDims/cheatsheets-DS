Tipos de JOIN

I

Un JOIN une filas de dos tablas según una condición (normalmente PK = FK). Lo que cambia entre tipos es qué filas sin pareja se conservan.

JOINDevuelve
INNER JOINsolo filas con pareja en ambas tablas
LEFT JOINtodas las de la izquierda + pareja (o NULL)
RIGHT JOINtodas las de la derecha + pareja (o NULL)
FULL OUTER JOINtodas de ambos lados; rellena con NULL
CROSS JOINproducto cartesiano (todas × todas)
CuándoUsa LEFT JOIN para "todos los clientes, hayan comprado o no" y filtra WHERE p.id IS NULL para hallar los que no tienen pareja.

INNER y LEFT

I
Aliasclientes c abrevia la tabla; con varios JOINs evita ambigüedad de columnas.

SELF JOIN

A

Una tabla unida consigo misma: típico para jerarquías (empleado → jefe).

TrucoCada uso necesita un alias distinto (e, j) para distinguir las dos "copias".
05

Agregación y grupos

GROUP BY · agregados
HAVING

GROUP BY y agregados

I

Colapsa filas en grupos y calcula un valor por grupo.

FunciónDevuelve
COUNT(*)nº de filas
SUM / AVGsuma / promedio
MIN / MAXmínimo / máximo
ReglaToda columna del SELECT que no esté en un agregado debe ir en el GROUP BY.

HAVING y orden lógico

I

WHERE filtra filas; HAVING filtra grupos ya agregados.

Orden de ejecuciónFROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. Por eso un alias del SELECT no se puede usar en el WHERE.

Subconsultas y CTE

recursiva

Subconsultas

I

Una consulta dentro de otra. Útil en WHERE, en el SELECT o como tabla derivada.

EXISTSWHERE EXISTS (SELECT 1 …) comprueba si hay filas relacionadas; suele ser más rápido que IN con muchos valores.

CTE (WITH)

A

Una tabla temporal con nombre para la consulta. Hace legible lo que una subconsulta anidada vuelve ilegible.

RecursivaWITH RECURSIVE recorre jerarquías (árbol de organigrama, lista de materiales) en una sola consulta.

OVER y PARTITION BY

A

Calculan sobre un "ventanal" de filas sin colapsarlas: a diferencia de GROUP BY, conservas cada fila y le añades el cálculo.

IdeaPARTITION BY = "grupos" de la ventana; ORDER BY dentro de OVER = orden para rankings y acumulados.

RANK · LAG · LEAD

A
FunciónPara
ROW_NUMBERnº correlativo único
RANKranking, salta tras empates
DENSE_RANKranking sin saltos
LAG / LEADvalor de la fila anterior/siguiente
Caso típicoLAG(ventas) OVER (ORDER BY mes) da la cifra del mes previo para calcular el crecimiento.

Diseño y datos con criterio

claves · índices
normalización · ACID

Claves PK y FK

I

PK (primary key) identifica cada fila de forma única; FK (foreign key) apunta a la PK de otra tabla y garantiza la integridad referencial.

Por quéLa FK impide insertar un pedido con un cliente_id que no existe, y bloquea borrar un cliente con pedidos (salvo ON DELETE CASCADE).

Normalización 1FN–3FN

A

Organizar el esquema para eliminar redundancia y anomalías al actualizar.

1FNvalores atómicos; sin listas ni grupos repetidos en una celda.
2FN1FN + cada columna depende de la PK completa (relevante en claves compuestas).
3FN2FN + sin dependencias transitivas (una columna no-clave no depende de otra no-clave).
EquilibrioNormaliza para integridad (OLTP); desnormaliza a propósito para velocidad de lectura en informes (OLAP / data warehouse).

Diagrama de relaciones (firma)

I

Tres tablas unidas por claves: cada pedido pertenece a un cliente (FK cliente_id) y referencia un producto (FK producto_id). Las líneas marcan la cardinalidad 1—∞.

PK clave primaria FK clave foránea 1—∞ uno a muchos
Cardinalidad1:1 una fila ↔ una fila · 1:N una fila ↔ muchas (lo común: cliente → pedidos) · N:M muchas ↔ muchas, que no se modela directo: se resuelve con una tabla puente (p. ej. pedido_producto con dos FK que apuntan a cada lado).

Índices

A

Estructura que acelera la búsqueda de filas: convierte un scan de toda la tabla en una búsqueda directa.

CosteCada índice acelera lecturas pero frena INSERT/UPDATE y ocupa espacio. Indexa columnas de WHERE, JOIN y ORDER BY frecuentes, no todas.

Vistas

I

Una consulta guardada con nombre que se usa como si fuera una tabla. No almacena datos: encapsula lógica.

Por quéSimplifica consultas repetidas y controla permisos (das acceso a la vista, no a las tablas base).

Transacciones · ACID

A

Una transacción agrupa varias sentencias en una unidad todo-o-nada: o se confirman todas (COMMIT) o ninguna (ROLLBACK).

A · Atomicidadtodo o nada; nada queda a medias.
C · Consistenciala BD pasa de un estado válido a otro.
I · Aislamientolas transacciones concurrentes no se pisan.
D · Durabilidadtras COMMIT, persiste aunque falle la luz.
CuándoImprescindible cuando varias escrituras deben cuadrar juntas (transferencias, reservas). Ante un error, ROLLBACK.

SERIAL · IDENTITY · UUID

I

Para claves autoincrementales. SERIAL es el clásico; GENERATED … AS IDENTITY es el estándar moderno recomendado.

PG vs genéricoMySQL usa AUTO_INCREMENT; en Postgres es SERIAL/IDENTITY. gen_random_uuid() viene en pgcrypto (PG 13+ ya lo trae integrado).

Arrays · rangos · enum

A

Postgres permite tipos que el SQL estándar no tiene.

ArraysIndexa con base 1: tags[1]. ANY() / @> prueban pertenencia; los ENUM se crean con CREATE TYPE mood AS ENUM(...).
03

JSONB

-> · ->> · @>
operadores

Anatomía JSONB (firma)

I

JSONB guarda JSON en binario: más rápido de consultar e indexable (a diferencia de JSON, que es texto crudo). Dada una columna datos JSONB con este documento, cada operador extrae una pieza distinta:

{ "cliente": "Ana", "items": ["sql", "psql"], "envio": { "pais": "PE" }, "pagado": true }
->devuelve JSON: datos->'envio' → otro objeto.
->>devuelve texto: datos->>'cliente' → Ana.
#>ruta anidada: datos#>'{envio,pais}'.
@>contiene: datos @> '{"pagado":true}'.
?¿existe la clave? datos ? 'envio'.
ReglaEncadena -> mientras navegas objetos y cierra con ->> cuando quieras el valor como texto comparable.

Consultar JSONB

A
CuándoUsa JSONB para datos semiestructurados o variables; si los campos son fijos y se consultan mucho, columnas normales siguen siendo mejores.

Tipos de índice

A

Postgres ofrece varios métodos; el correcto depende del dato y del operador.

TipoIdeal para

EXPLAIN ANALYZE

A

Muestra el plan real y los tiempos medidos: la herramienta nº 1 para diagnosticar consultas lentas.

Lee estoSeq Scan = recorre toda la tabla (sospechoso si es grande). Index Scan = usó el índice. Compara cost estimado con el tiempo real.

Secuencias

I

Generadores de números que respaldan a SERIAL. Útil compartir una secuencia entre tablas.

NotaTras carga masiva, reajusta con setval('folio', max_id) para evitar choques de PK.

UPSERT (ON CONFLICT)

A

Inserta, y si la fila ya existe (choca con una clave única), actualiza en vez de fallar.

EXCLUDEDEs la fila que se intentó insertar. Usa DO NOTHING para ignorar duplicados sin error.

Backups y mantenimiento

pg_dump · restore
VACUUM

pg_dump · restaurar

I

Respaldo lógico desde la terminal del sistema. El formato custom (-Fc) es comprimido y flexible.

NotaCon -Fp (texto plano) restauras con psql -d base -f archivo.sql. Para todo el clúster: pg_dumpall.

VACUUM y mantenimiento

A

Postgres marca filas borradas como "muertas"; VACUUM recupera ese espacio y refresca estadísticas.

CuándoEl autovacuum lo hace solo en segundo plano; ejecuta ANALYZE manual tras cargas masivas para que el planificador tenga estadísticas frescas.
