¿Qué es Git y qué es GitHub?

B

Git es el sistema de control de versiones que corre en tu máquina: guarda fotos (commits) de tu proyecto a lo largo del tiempo y te deja volver atrás, ramificar y combinar. GitHub es un servicio en la nube para alojar repos Git y colaborar (pull requests, issues, CI).

Repositoriola carpeta del proyecto + su historial completo (en .git/).
Commituna instantánea con mensaje; la unidad de cambio. Tiene un hash único.
Rama (branch)una línea de trabajo paralela. main es la principal.
Remotouna copia del repo en un servidor (GitHub); por defecto se llama origin.
La idea claveGit es distribuido: cada copia tiene el historial entero. Puedes trabajar sin conexión y sincronizar después.

Configurar & empezar

B

Configura tu identidad una vez (global). Luego inicia un repo nuevo con init o copia uno existente con clone.

SSH vs HTTPSConfigura una clave SSH en GitHub para empujar sin escribir contraseña cada vez (ssh-keygen + añadir la clave pública en Settings).

Las tres zonas — el corazón de Git

B

Un archivo viaja por tres zonas. Entender este flujo explica casi todos los comandos: add mueve a la izquierda→staging; commit de staging→repo.

El ciclo básico

B

Editar → revisar → preparar → confirmar. El 90% del trabajo diario son estos cuatro comandos.

Buen mensajeEn imperativo y conciso: "Añade validación de email", no "añadí cosas". La 1.ª línea ≤ 50 caracteres; detalle en el cuerpo si hace falta.

status & diff

B

status resume el estado; diff muestra qué líneas cambiaron exactamente.

log & historial

I

Recorre el historial de commits. La forma --oneline --graph es la más útil para ver ramas de un vistazo.

.gitignore

B

Lista de patrones que Git no debe rastrear: dependencias, secretos, binarios, entornos. Va en la raíz del repo.

Ya lo rastreé.gitignore no afecta a lo ya rastreado. Sácalo del índice con git rm --cached <archivo> y haz commit.

Qué es una rama — y cómo se ve

B

Una rama es un puntero móvil a un commit. Creas una rama para una tarea (feature/login), trabajas aislado y luego la fusionas (merge) de vuelta a main. Así el historial se ramifica y se vuelve a unir:

git log --oneline --graph
HEADpuntero a "dónde estás": normalmente la punta de la rama actual.
Por qué ramificaraíslas trabajo en curso de main (que se mantiene estable y desplegable).

Crear & cambiar de rama

B
switch vs checkoutgit switch (moderno) cambia de rama; git restore recupera archivos. Antes ambas cosas las hacía el sobrecargado checkout.

Merge & sus tipos

I

Estando en la rama destino (main), fusionas la otra hacia ella.

Fast-forwardsi main no avanzó, solo mueve el puntero. Historial lineal.
Merge commitsi ambas avanzaron, crea un commit que une las dos historias.

Trabajo remoto (GitHub)

sincronizar con
el equipo

clone & remote

B

El remoto por defecto se llama origin. clone ya lo configura; en un repo local lo añades a mano.

fetch · pull · push

I

fetch baja cambios sin aplicarlos; pull = fetch + merge; push sube los tuyos.

Rechazado al pushSi el remoto tiene commits que no tienes, Git rechaza el push. Haz git pull (o --rebase), resuelve y vuelve a empujar. Nunca --force en ramas compartidas.

Pull requests — el flujo de GitHub

I

Un pull request (PR) propone fusionar tu rama en main y abre un espacio para revisión, comentarios y CI antes de integrar. El flujo estándar de equipo:

Squash mergeSquash aplasta todos los commits del PR en uno solo sobre main: historial limpio. Muy usado en equipos.

Deshacer sin romper nada

el botón de
marcha atrás

restore & reset

I

restore descarta cambios en archivos; reset mueve la punta de la rama. Las tres variantes de reset según qué conservan:

Cuidado con --hardDestruye cambios no commiteados sin red de seguridad. Si dudas, --soft o git stash primero.

revert & amend

I

revert es el modo seguro de deshacer en ramas compartidas: crea un commit nuevo que invierte otro, sin reescribir la historia. amend corrige el último commit.

amend ya empujado--amend reescribe el commit: si ya lo empujaste a una rama compartida, evita reescribir. Úsalo solo en commits locales.

stash & reflog — la red de seguridad

A

stash guarda cambios a medio hacer en un cajón para cambiar de rama sin commitear. reflog registra cada movimiento de HEAD: tu salvavidas para recuperar commits "perdidos" tras un reset.

Casi nada se pierdeAunque hagas un reset --hard de más, el commit sigue en el reflog unos 90 días. Busca su hash con git reflog y recupéralo.

merge vs rebase

A

Ambos integran cambios. Merge conserva la historia real (con commit de unión); rebase reescribe tus commits encima de la otra rama para una historia lineal.

La regla de oro del rebaseNo rebases ramas públicas/compartidas: reescribir commits que otros ya tienen rompe su historial. Rebasa solo tu trabajo local.

Resolver conflictos

I

Pasa cuando dos ramas tocan las mismas líneas. Git marca el choque; tú eliges qué queda, borras los marcadores y confirmas.

Tags & releases

I

Un tag marca un punto del historial (típicamente una versión) con un nombre estable. Combinado con GitHub Releases, publica versiones descargables.

SemVervMAYOR.MENOR.PARCHE: MAYOR rompe compatibilidad, MENOR añade features, PARCHE corrige bugs. v2.1.3.
