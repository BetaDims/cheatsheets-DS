01

Componentes & JSX

B · fundamentos

Qué es un componente

B

Una función que devuelve JSX (la UI). Su nombre va en PascalCase. Se reutiliza como una etiqueta propia y se compone con otros componentes.

Por qué: componentes pequeños y con un solo propósito se prueban, reutilizan y razonan mejor. Empieza dividiendo la UI en piezas.

El árbol de componentes

B

React renderiza una jerarquía: un componente raíz que contiene otros. Los datos bajan por props; los eventos suben por callbacks.

Flujo de datos: es unidireccional. El estado vive arriba y desciende; para cambiarlo desde abajo se pasa una función como prop.

Reglas de JSX

B
  • Devuelve un solo nodo raíz; usa <>…</> (Fragment) si hace falta.
  • Atributos en camelCase: className, onClick, htmlFor.
  • Expresiones JS entre llaves: {valor}.
  • Cierra siempre las etiquetas: <img />.
Por qué llaves: dentro del JSX, todo lo que sea JavaScript (variables, llamadas, ternarios) va en { }; el resto es marcado literal.

Props: datos hacia abajo

B

Las props son los argumentos del componente. Son de solo lectura: nunca las mutes. Desestructura y da valores por defecto.

Regla: los datos fluyen del padre al hijo. Si un hijo necesita cambiar algo, recibe una función del padre y la llama (no toca la prop).

children & composición

I

La prop especial children deja anidar contenido. Componer (envolver) es preferible a heredar.

Cuándo: usa children para layouts, tarjetas y modales genéricos: el envoltorio no necesita saber qué contiene.

Convenciones

B

En los snippets, así = reemplázalo por tu valor.

prop nombre de una propiedad del componente
state variable de estado (la que devuelve useState)
handler función manejadora, ej. handleClick
deps array de dependencias de un hook
02

Estado & eventos

B → I · interactividad

useState: memoria del componente

B

Devuelve un par: el valor y su setter. Llamar al setter re-renderiza el componente con el nuevo valor.

Por qué no una variable normal: una variable local se pierde en cada render. El estado lo conserva React entre renders y dispara la actualización de la UI.

Manejar eventos

B

Pasa la función, no la llames: onClick={handler}, no onClick={handler()}.

Atajo: usa una flecha inline onClick={() => fn(id)} cuando necesites pasar argumentos.

Actualizar según el estado previo

I

Si el nuevo valor depende del anterior, pasa una función al setter. Evita errores cuando hay varias actualizaciones seguidas.

Para objetos y arrays, crea una copia nueva (inmutabilidad); no mutes el estado actual.

Por qué inmutable: React compara referencias para decidir si re-renderiza; mutar el mismo objeto no cambia la referencia y la UI no se actualiza.

Renderizado condicional

B
Ojo con &&: si la izquierda es 0, React imprime el 0. Convierte a booleano: {list.length > 0 && …}.

Listas con map & key

B

Renderiza colecciones con .map(). Cada elemento necesita una key estable y única.

Por qué la key: identifica cada elemento entre renders para reordenar/actualizar sin recrear todo. Usa el id del dato, no el índice del array (cambia al insertar o borrar).
03

Efectos & refs

I · sincronizar con el exterior

useEffect: efectos secundarios

I

Sincroniza el componente con sistemas externos (red, timers, suscripciones, DOM). Corre después del render.

Cuándo NO usarlo: no para transformar datos para el render (hazlo en el cuerpo) ni para responder a eventos de usuario (hazlo en el handler). useEffect es para sincronizar, no para "hacer cosas".

Limpieza & array de dependencias

I

Devuelve una función de limpieza para deshacer el efecto (timers, listeners) antes de re-ejecutar o al desmontar.

depsCuándo corre
[]una vez, al montar
[a, b]al montar y si a o b cambian
sin arraytras cada render (raro)
Regla: incluye en deps toda variable reactiva usada dentro. Omitirla causa bugs por valores "viejos" (stale closures).

useRef: valor que no re-renderiza

I

Una "caja" mutable (.current) que persiste entre renders sin provocar re-render. Dos usos: acceder al DOM y guardar un valor mutable.

ref vs state: usa state para datos que se ven en la UI; ref para lo que no debe disparar render (un id de timer, el nodo DOM, un valor previo).
04

Formularios controlados

I · entrada de usuario

Input controlado

I

El estado es la única fuente de verdad: el input lee su value del estado y lo actualiza en onChange.

Varios campos: usa un objeto en el estado y actualízalo por name: setForm(f => ({...f, [e.target.name]: v})).

Submit & validación

I
Siempre e.preventDefault() en el submit para que el navegador no recargue la página.

Levantar el estado (lifting state up)

I

Cuando dos hermanos necesitan compartir datos, sube el estado al ancestro común. El padre guarda el estado y pasa valor + setter a los hijos.

Por qué: mantiene una sola fuente de verdad. Si la compartición se vuelve incómoda (prop drilling), pasa a Context (sección 07).
05

Hooks avanzados

I → A · lógica reutilizable

Reglas de los hooks

I
  • Llámalos solo en el nivel superior: nunca dentro de if, bucles o funciones anidadas.
  • Solo desde componentes o desde otros custom hooks.
  • El nombre de un hook empieza por use.
Por qué: React identifica cada hook por su orden de llamada. Llamarlos condicionalmente desordena ese registro y rompe el estado. El plugin ESLint react-hooks lo vigila por ti.

useContext: evitar prop drilling

I

Comparte datos con un subárbol sin pasar props nivel a nivel. Crea el contexto, envuelve con Provider y consúmelo con useContext.

Cuándo: datos "globales" de baja frecuencia (tema, usuario, idioma). Para estado que cambia mucho, Context puede re-renderizar de más: ahí piensa en Redux (sección 07).

useReducer: estado complejo

A

Centraliza la lógica de transición de estado en una función pura reducer(state, action). Mejor que varios useState entrelazados.

Cuándo: varias subpartes del estado que cambian juntas, o transiciones con muchos casos. El reducer es testeable de forma aislada.

useMemo / useCallback

A

useMemo memoriza un valor calculado; useCallback memoriza una función. Ambos solo recalculan si cambian sus deps.

Cuándo memorizar: NO por defecto. Úsalo si (a) el cálculo es realmente caro, o (b) pasas el valor/función a un hijo envuelto en React.memo y quieres evitar su re-render. Memorizar de más añade complejidad sin ganancia.

Custom hooks

A

Extrae lógica con estado a una función useAlgo() reutilizable. Puede usar otros hooks y devolver lo que quieras.

Por qué: comparten lógica, no estado: cada componente que llama al hook obtiene su propia copia. Ideal para fetch, formularios, suscripciones, etc.
06

React Router

I · navegación SPA

Definir rutas

I

Mapea URLs a componentes. Instala react-router-dom y envuelve la app en un router.

path='*' captura cualquier ruta no encontrada: úsalo para tu página 404.

Link, params & navegación

I

Navega con <Link> (sin recargar). Lee parámetros de la URL con useParams y navega por código con useNavigate.

Link vs a: <Link to='/x'> cambia la vista sin recargar la página; un <a> normal recarga toda la app.

Rutas anidadas & Outlet

I

Una ruta padre con layout común renderiza a sus hijas donde pongas <Outlet />.

index marca la hija por defecto cuando la URL coincide exactamente con el padre.
07

Estado global & datos

I → A · escalar el estado

Context vs Redux Toolkit

A

Dos formas de compartir estado global. Compara y elige según frecuencia de cambios y tamaño de la app.

Context + Reducer: cero dependencias, ideal para estado global sencillo y de cambios poco frecuentes (tema, auth). Puede re-renderizar de más si crece.

Redux Toolkit: slice + hooks

A

Un slice agrupa estado + reducers. RTK usa Immer, así que puedes "mutar" el borrador con seguridad.

Cuándo Redux: apps grandes, estado compartido y cambiante, necesidad de DevTools/time-travel y lógica predecible. Si no, Context basta.

Fetching de datos

I

Patrón clásico con useEffect: estados de datos, cargando y error.

En serio: para producción usa una librería (TanStack Query, RTK Query): te dan caché, reintentos y estados de carga sin reinventarlos. El flag active evita actualizar tras desmontar.

Un vistazo a Next.js

A · React en el servidor

Qué aporta Next.js

A

Un framework sobre React con enrutado por archivos, renderizado en servidor (SSR), Server Components, optimización y backend integrado.

  • SSR/SSG: HTML listo desde el servidor → mejor SEO y primera carga.
  • App Router: rutas según carpetas en app/.
  • Server Components por defecto: menos JS al cliente.

App Router: rutas por carpetas

A

Cada carpeta en app/ con un page.jsx es una ruta. [id] es un segmento dinámico; layout.jsx envuelve a sus hijas.

Convención = configuración: no declaras rutas en código; la estructura de carpetas es el mapa de rutas.

Server vs Client Components

A

En App Router todo es Server Component por defecto (corre en el servidor, sin JS al cliente). Añade 'use client' arriba para los que necesiten estado, efectos o eventos.

Regla práctica: deja servidor por defecto (datos, peso menor) y marca como cliente solo las hojas interactivas del árbol.

Datos & SSR

A

En un Server Component puedes usar async/await directamente: el fetch corre en el servidor antes de enviar el HTML.

Por qué importa: el dato llega ya renderizado (sin spinner inicial ni waterfall en el cliente), con mejor SEO y menos JavaScript enviado.
