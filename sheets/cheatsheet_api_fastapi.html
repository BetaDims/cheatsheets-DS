¿Qué es una API?

B

Una API (Application Programming Interface) es un contrato que deja que un programa hable con otro sin saber cómo está hecho por dentro. Tú envías una petición y recibes una respuesta, siempre con las mismas reglas.

Metáforael camarero: pides del menú (la API), la cocina (el servidor) prepara y te trae el plato. No entras a la cocina.
Clientequien pide (app, web, móvil, otro servidor).
Servidorquien responde y guarda los datos.
Por qué importaLas APIs desacoplan: el front no necesita saber de la base de datos. Y se reutilizan: la misma API sirve a la web, al móvil y a terceros.

REST & recursos

B

REST es un estilo para diseñar APIs sobre HTTP. Todo gira en torno a recursos (cosas: usuarios, libros) con una URL cada uno, sobre los que aplicas verbos HTTP.

URL del recursoRepresenta
/booksla colección de libros
/books/12el libro con id 12
/books/12/tagslos tags de ese libro
ReglaNombres en plural y sustantivos, no verbos: /books, nunca /getBooks (el verbo lo pone HTTP).

JSON — el idioma común

B

El formato en que viajan los datos: texto con pares clave-valor, legible para humanos y máquinas. Es el cuerpo típico de peticiones y respuestas.

El método dice qué quieres hacer con el recurso. Mapean directamente al CRUD: Crear, Read, Update, Delete.

PATCHUpdatemodificar un campoPATCH /books/12

Seguro vs idempotente

I

Dos propiedades que guían qué método usar y cómo reintentar sin romper nada.

Segurono cambia datos. Solo GET.
Idempotenterepetirlo da el mismo resultado: GET, PUT, DELETE.
No idemp.POST dos veces crea dos recursos.

PUT vs PATCH

I

Ambos actualizan, pero no igual:

PUTmanda el recurso completo; lo que omitas se borra.
PATCHmanda solo los campos a cambiar.
PrácticaMuchas APIs (y el curso de FastAPI) usan PUT para todas las actualizaciones por simplicidad.

Las cinco familias

B

El primer dígito te dice la categoría; el número exacto, el detalle. 2xx = bien, 4xx = culpa del cliente, 5xx = culpa del servidor.

CódigoSignificado
200 OKtodo bien (GET, PUT)
201 Createdrecurso creado (POST)
204 No Contentok, sin cuerpo (DELETE)
301 / 302redirección
304 Not Modifiedusa tu caché
CódigoSignificado
400 Bad Requestpetición mal formada
401 Unauthorizedfalta o falla el login
403 Forbiddenlogueado, sin permiso
404 Not Foundno existe el recurso
422 Unprocessablevalidación falló
500 / 503error del servidor
401 vs 403401 = "no sé quién eres" (autentícate). 403 = "sé quién eres, pero no puedes". 422 es el que más verás en FastAPI: los datos no pasan la validación.

El ciclo de una petición — qué pasa entre el cliente y tu función

I

Cuando llega una petición, FastAPI la pasa por una cadena: el servidor ASGI, el middleware, el router, las dependencias y Pydantic (que valida la entrada) — y solo entonces ejecuta tu función. A la vuelta, Pydantic serializa la respuesta. Si la validación falla, FastAPI corta y responde 422 sin llegar a tu código.

Lo claveLa validación va antes de tu función y la serialización después: tu código solo ve datos ya tipados y devuelve objetos, no JSON a mano. Un raise HTTPException(...) en cualquier punto corta la cadena y vuelve directo al cliente.

Path & query parameters

I

El parámetro de la ruta se declara con tipo; los de query son los argumentos extra de la función (con valor por defecto).

Validación gratisSi llaman /books/abc, FastAPI responde 422 solo: abc no es int.

Modelos Pydantic (el body)

I

Un modelo Pydantic describe el JSON que entra o sale. Field añade reglas; FastAPI valida y documenta solo.

Hashing & JWT

A

Nunca guardes contraseñas en claro: hashea con bcrypt. Tras login, emite un JWT firmado que el cliente envía como Bearer.

CuidadoEl SECRET va en variable de entorno. Un JWT no se cifra, se firma: no metas datos sensibles en su cuerpo.
